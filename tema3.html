Requisitos previos

Instala Git y crea una cuenta en GitHub. Verifica tu instalación:

$ git --version
$ git config --global user.name "Tu Nombre"
$ git config --global user.email "tu@email.com"

1. ¿Qué es un sistema de control de versiones?

Un Sistema de Control de Versiones (VCS) registra los cambios de tus archivos en el tiempo, permite volver a estados anteriores, colaborar con seguridad y auditar quién cambió qué y cuándo.

  • Historial completo y recuperable.
  • Trabajo en paralelo sin sobrescribir a otros.
  • Integraciones con CI/CD, revisiones y seguridad.

2. Sistemas de control de versiones usados en empresas

Distribuidos

  • Git (estándar de la industria)
  • Mercurial (HG)

Centralizados / Mixtos

  • Subversion (SVN)
  • Perforce (Helix Core) — frecuente en juegos/diseño

Plataformas de hosting Git

  • GitHub, GitLab, Bitbucket, Azure Repos.

3. ¿Qué es Git?

Git es un VCS distribuido: cada clon contiene el historial completo. Es rápido, seguro (hashes SHA), y permite ramas/merges eficientes.

4. ¿Qué es GitHub?

GitHub es una plataforma para alojar repositorios Git con funciones de colaboración: issues, Pull Requests (PR), revisiones, CI/CD, gestión de equipos y seguridad (Actions, Dependabot, etc.).

5. ¿Cómo subo un repositorio a GitHub? (paso a paso)

  1. Crea un repo vacío en GitHub (sin README/ .gitignore para evitar conflictos).
  2. En tu carpeta local inicializa Git: 
    $ git init
$ echo "# Mi Proyecto" > README.md
$ git add .
$ git commit -m "feat: primer commit"
  3. Conecta el remoto y empuja main: 
    $ git remote add origin https://github.com/usuario/mi-proyecto.git
$ git branch -M main
$ git push -u origin main
  4. Comprueba en GitHub que aparecen los archivos y el historial.

Si el repo remoto ya tenía commits, primero haz git pull --rebase origin main antes del push.

6. ¿Cómo se crean ramas y cómo funciona en la industria?

Crear y trabajar en una rama

# crear y moverte a una rama
$ git checkout -b feature/login
# trabajar y commitear
$ git add .
$ git commit -m "feat(login): formulario y validaciones"
# publicar la rama en remoto
$ git push -u origin feature/login

Flujos comunes

  • Feature Branch: rama por tarea/issue → PR a main.
  • GitFlow: ramas develop, release, hotfix.
  • Trunk-Based: commits pequeños a main con feature flags.

7. ¿Cómo clono un repositorio?

$ git clone https://github.com/organizacion/proyecto.git
$ cd proyecto
$ git remote -v   # verifica remotos

Para clonar con SSH (recomendado en empresas): git@github.com:organizacion/proyecto.git

8. ¿Cómo es la mecánica para hacer pull?

git pull trae cambios del remoto y los integra a tu rama actual (fetch + merge). En equipos suele preferirse rebase para historial lineal.

# opción merge (por defecto)
$ git pull origin main
# opción rebase (historial más limpio)
$ git pull --rebase origin main

9. ¿Qué es un Pull Request y cómo es la dinámica?

Un Pull Request (PR) es una solicitud para integrar cambios desde tu rama hacia otra (p. ej., main). La dinámica típica en empresas:

  1. Empujas tu rama al remoto.
  2. Creas el PR en la plataforma (título claro, descripción, capturas, checklist y enlace a issue).
  3. Revisores dan feedback y aprueban.
  4. CI debe pasar (tests, lint, build).
  5. Se realiza merge o squash & merge y se elimina la rama.

10. ¿Qué es merge?

Merge combina el historial de dos ramas. Puede crear un commit de merge si hay divergencia.

$ git checkout main
$ git merge feature/login   # une la rama al main

Alternativa: rebase reescribe tu rama encima de otra para un historial lineal (git rebase main).

11. ¿Cómo se fuerza un push y en qué casos se usa?

Un push forzado (--force-with-lease) sobrescribe el remoto con tu historial local. Úsalo solo cuando reescribiste historial (rebase/fixup) en tu rama propia, nunca en main compartida.

# tras rebase interactivo, publicar cambios reescritos de forma segura
$ git push --force-with-lease origin feature/login

Riesgo: puede perder cambios ajenos si no verificas. --force-with-lease reduce el riesgo comparado con --force.

12. ¿Qué posibles conflictos pueden ocurrir en una rama?

  • Edición en la misma línea de un archivo por dos ramas.
  • Borrado vs. modificación del mismo archivo.
  • Conflictos binarios (imágenes, builds).
  • Conflictos al rebase (reaplicar commits sobre base nueva).
# resolver conflictos
$ git status                 # ve archivos en conflicto
$ git add archivo_resuelto   # marca como resuelto
$ git rebase --continue      # o git commit si estabas en merge

Consejos pro

Commits pequeños

Facilitan la revisión y el revert.

Mensajes claros

Usa prefijos: feat:, fix:, docs:, etc.

Pull frecuente

Reduce conflictos y sorpresas al integrar.

Tarea final: Subir el repositorio seguimiento

  1. En GitHub, crea el repo seguimiento (público o privado).
  2. En tu PC: 
    $ mkdir seguimiento && cd seguimiento
$ git init
$ echo "Proyecto de seguimiento" > README.md
$ git add .
$ git commit -m "chore: init seguimiento"
  3. Conecta y empuja a GitHub: 
    $ git remote add origin https://github.com/usuario/seguimiento.git
$ git branch -M main
$ git push -u origin main
  4. Crea una rama feature/tarea-1, añade un archivo tarea1.md y sube un PR.

Entregable: enlace al repositorio y al PR.
